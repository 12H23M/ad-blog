## "리나, 이거 왜 안 돼?"

새벽 1시. 봇이 거래를 실행했는데 DB에 기록이 안 남는다. 로그를 뒤져도 에러가 안 보인다. 머리를 싸매고 30분째 storage.py를 읽고 있었다.

디스코드를 열었다. 리나한테 물어봤다.

> "리나, crypto_trades 테이블에 저장이 안 돼. 에러도 없어."

5초 후 리나가 코드를 분석하더니 답했다.

> "storage.py에서 테이블명이 `trades`로 되어 있는데, 실제 Supabase 테이블은 `crypto_trades`야."

**오타.** 30분 동안 삽질한 게 테이블명 오타 하나였다.

이게 내가 AI와 함께 개발하는 방식이다. 혼자 끙끙대는 대신 **즉시 물어보고, 즉시 답을 듣고, 즉시 고친다.**

## 리나가 누군데?

리나는 내 AI 비서다. [OpenClaw](https://openclaw.ai)라는 플랫폼 위에서 돌아가는데, 쉽게 말하면 **디스코드에 상주하는 AI 개발 파트너**다.

일반적인 ChatGPT랑 뭐가 다르냐면:

- **24시간 디스코드에 있다.** 새벽 3시에 에러 나도 바로 물어볼 수 있다.
- **내 프로젝트를 안다.** 코드, 아키텍처, 지난 대화를 기억한다.
- **직접 코드를 수정한다.** 파일을 읽고, 편집하고, 커밋하고, 푸시까지 한다.
- **크론잡으로 자동 점검한다.** 4시간마다 봇 상태 확인하고, 죽은 봇 재시작해준다.
- **서브에이전트를 띄운다.** 큰 작업은 별도 Claude Code를 실행해서 병렬로 처리한다.

실제로 이 블로그 사이트도 리나가 만들었다. "코인 블로그로 바꿔줘"라고 했더니 Astro 프로젝트를 리셋하고, 다크 테마 적용하고, i18n 세팅하고, 첫 포스트 작성하고, 배포까지 한 번에 해줬다.

## Claude Code: 무거운 작업은 이 녀석한테

리나가 직접 하기엔 긴 작업 — 예를 들어 "블로그 전체 디자인 리뉴얼" 같은 건 **Claude Code**한테 넘긴다.

Claude Code는 터미널에서 돌아가는 코딩 에이전트다. 프롬프트를 주면 알아서 파일을 읽고, 코드를 짜고, 테스트하고, 커밋한다.

실제 워크플로우:
```
리나가 프롬프트 파일 작성 (/tmp/blog-rebuild.md)
→ Claude Code 에이전트 백그라운드 실행
→ 15분 후 작업 완료 알림
→ 리나가 결과 확인 후 푸시
```

이 블로그의 다크 테마, 컴포넌트 9개, 페이지 구조 전부가 이런 식으로 만들어졌다. 나는 "이렇게 해줘"라고 말하고, AI가 실행한다.

근데 항상 완벽하진 않다. Claude Code가 10분 타임아웃에 signal 15로 죽은 적도 있고, OAuth 토큰이 만료돼서 실패한 적도 있다. 그때는 리나가 직접 마무리한다.

## OpenCode: 가벼운 작업에 빠르게

OpenCode도 비슷한 코딩 에이전트인데, 좀 더 가볍다. 파일 하나 수정하거나, 버그 하나 잡을 때 쓴다.

다양한 AI 모델을 쓸 수 있는 게 장점이다. GPT-4, Claude, 오픈소스 모델까지. 작업 성격에 따라 모델을 바꿀 수 있다.

## 실제로 AI가 한 일들

OWL 프로젝트에서 AI가 직접 기여한 것들:

**리나가 한 것:**
- `storage.py` 테이블명 버그 발견 (`trades` → `crypto_trades`)
- `close_trade()` 잘못된 전략 필터 추가 제안
- 수수료 반영 누락 발견 + 3개 청산 경로에 모두 적용
- 일일 요약 시스템 설계 + 구현
- 15개 전략 백테스트 전부 실행 + 결과 분석
- OKX `create_order` 응답이 체결가/수량 신뢰 불가 → `fetch_order` 재시도 로직 제안
- 이 블로그 전체 (디자인, 포스트, 배포)

**Claude Code가 한 것:**
- 블로그 다크 테마 + 컴포넌트 전체 구현 (850줄+)
- 디자인 고도화 (그라디언트 오브, 카드 컬러바, 스크롤바)
- 스크리너 포스트 한/영 동시 작성

**내가 한 것:**
- 전체 아키텍처 결정
- 전략 아이디어 제시 ("컨센서스 투표 방식으로 해보자")
- 실전 배포 판단 ("이거 라이브에 올려도 되겠다")
- 리스크 파라미터 결정 ("거래당 1%, 일일 3%")
- AI가 제안한 것 중 뭘 수용하고 뭘 버릴지 판단

비율로 치면 **코드의 80%는 AI가 짰다.** 하지만 **판단의 100%는 내가 했다.** AI는 도구고, 방향을 정하는 건 사람이다.

## AI 코딩의 현실

장밋빛만은 아니다. 솔직한 후기:

**좋은 점:**
- 개발 속도가 미친다. 혼자 3개월 걸릴 걸 3주에 했다.
- 새벽에 혼자 삽질 안 해도 된다.
- 코드 리뷰가 즉시 된다.
- 지루한 반복 작업(백테스트 15개 돌리기)을 맡길 수 있다.

**나쁜 점:**
- AI가 "자신 있게" 틀린 답을 할 때가 있다. 맹신하면 위험.
- 긴 프롬프트를 주면 중간에 죽거나 잘못된 방향으로 갈 때가 있다.
- 코드는 짜주는데, 왜 이렇게 짰는지 이해 못하면 유지보수가 지옥이 된다.
- "알아서 해줘"는 위험하다. 구체적으로 지시할수록 결과가 좋다.

**가장 큰 교훈:**
AI는 **주니어 개발자**와 비슷하다. 시키면 빠르게 하는데, 뭘 시킬지는 시니어(나)가 정해야 한다. 방향 없이 "알아서 해봐"라고 하면 엉뚱한 곳에서 삽질한다.

## 미래: AI와 함께 트레이딩

지금 OWL에는 **AGI Think**라는 모듈이 있다. Claude API를 직접 호출해서 시장 상태를 분석하고, regime.json에 결과를 쓴다. 4시간마다 "지금 시장이 어떤 상태인지" AI가 판단한다.

다음 단계는 AI가 단순 분석을 넘어서 **전략 교체까지 자동으로** 하는 것이다. "지금 횡보장이니까 그리드 전략 켜고, 추세 전략은 줄여"를 AI가 판단하고 실행한다.

사람이 하는 일은 점점 줄어들고 있다. 좋은 건지 무서운 건지는 아직 모르겠다.
